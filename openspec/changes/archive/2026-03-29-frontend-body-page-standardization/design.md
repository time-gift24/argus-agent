## Context

当前前端已经有固定的全局壳层（`App.vue` 的 sidebar + top nav），但各个 `views` 内部的 body 结构仍然各自为战：`ProviderEditView` 是独立页面，`McpConfigsView` 仍通过弹窗承载创建/编辑，其他页面多数只有标题，没有统一的面包屑和页面级操作区。结果是用户进入页面后很难快速判断自己所处层级，复杂表单也容易在弹窗里变得拥挤。

这次变更只涉及前端信息架构与页面结构，不修改后端接口，也不引入新的 UI 库。已有的登录弹窗、删除确认弹窗仍然是合理的防呆场景，需要保留。

## Goals / Non-Goals

**Goals:**
- 为所有具体 route body 页面提供统一的头部骨架，固定在页面左上角展示面包屑
- 将高信息密度的创建/编辑流程收敛到独立页面，减少业务弹窗
- 让 Provider 与 MCP 的编辑页遵循一致的结构：页面头部、表单分区、测试反馈、提交操作
- 在不改后端 API 的前提下完成路由、视图和共享组件的整理

**Non-Goals:**
- 不重做全局 sidebar / top nav 视觉设计
- 不新增后端 CRUD、测试或权限接口
- 不处理移动端专属布局，本次以桌面端体验为准
- 不把登录确认、删除确认这类保护性弹窗一并移除

## Decisions

### D1: 引入共享的 body 页面壳，而不是在每个 view 中手写头部
采用一个共享页面壳组件（例如 `PageBodyShell`）来承载面包屑、标题、说明、副操作和主操作插槽。这样可以把“左上角固定面包屑 + 统一标题区”的约束变成结构性约束，而不是依赖每个页面自行实现。

备选方案是继续在每个 view 内手写头部。这个方案初期修改快，但会复制大量 Tailwind/TinyVue 结构，后续任何一个视觉调整都需要逐页回改，因此不采用。

### D2: “固定位置”的含义定义为 body 头部在内容区内 sticky，而不是浏览器级悬浮
页面面包屑与头部应固定在内容区左上角，并在页面内部滚动时保持可见，但不覆盖全局 top nav。实现上使用主内容区内的 sticky header，更符合现有 `App.vue` 壳层结构，也避免与顶部导航产生双层悬浮冲突。

备选方案是完全静态头部。它实现更简单，但一旦页面内容较长，用户滚动后会失去导航上下文，因此不采用。

### D3: 创建/编辑流程优先使用独立路由页面
Provider 已经有独立编辑页，这次保持该模式并做页面结构升级。MCP 列表页则新增 `/mcp/new` 和 `/mcp/:id/edit`，列表页只负责浏览、测试、进入编辑和删除确认，不再承载复杂表单弹窗。

备选方案是保留弹窗，仅放大尺寸或改成抽屉。这个方案虽然改动表面更小，但仍然会把多 transport 表单、JSON 输入和测试反馈挤在覆盖层里，不能真正解决信息架构问题，因此不采用。

### D4: 弹窗只保留给防呆和保护性场景
登录拦截、删除确认等短链路、高风险操作继续使用弹窗；业务录入、复杂编辑、信息浏览改为页面内完成。这样既符合“尽可能减少弹窗”的设计约束，也能减少用户在遮罩层和主页面之间反复切换。

### D5: LLM 与 MCP 编辑页共享相同的信息分区模型
两个编辑页都采用“头部摘要 + 基础配置区 + 连接测试/校验区 + 提交操作区”的结构。Provider 编辑页延续现有 `test-config` 流程；MCP 编辑页复用现有 CRUD 与测试接口，并明确说明编辑已有配置时敏感字段的保留规则。

备选方案是按各自业务独立设计页面。虽然灵活，但会继续放大两个配置页的交互差异，不利于形成一致的产品语言，因此不采用。

## Risks / Trade-offs

- **[多页面同时整改范围较大]** → 先抽共享页面壳，再逐个页面迁移，减少样式回归面
- **[sticky 头部可能与现有内容顶部间距冲突]** → 统一由页面壳负责顶部留白和 `top` 偏移，避免每个页面单独计算
- **[MCP 编辑页涉及敏感字段]** → 在页面文案中明确“留空表示保留现有 headers/env”，测试能力与保存语义分开设计
- **[桌面优先可能留下小屏表现差异]** → 本次 spec 明确移动端不在范围内，避免实现阶段产生额外设计债务

## Migration Plan

1. 新增共享的 body 页面壳组件，并迁移现有具体页面到统一头部结构
2. 保持 Provider 路由不变，重构其列表页与编辑页的 body 布局
3. 为 MCP 增加独立的新建/编辑路由，将列表页中的创建/编辑弹窗迁移为页面导航
4. 仅保留登录和删除确认等保护性弹窗，移除业务录入型弹窗
5. 完成后通过前端构建和相关页面手工回归验证主要桌面端流程
